# QCchem v0.3 Platform Design

## Goal

把 QCchem 从单次量子化学运行器提升为一个可组织研究活动的平台雏形，同时保持现有 run/result/benchmark 可信度链路不被打散。

## Design Summary

本轮以现有 `RunResult` 为中心扩展四层平台能力：

1. `backend capability + execution policy`
   统一描述 backend 能力矩阵、runtime-ready/mitigation-ready 状态，以及 benchmark/publication/hardware-ready 等策略默认值。

2. `study / benchmark / registry`
   让多个 run 可以被组织成 study，并把 benchmark suite 落成正式 registry 与 aggregate artifact。

3. `scan workflow`
   把 1D PES scan 作为 study 的特化形式，每个扫描点落独立 run artifact，同时输出 scan table 与 study 级总结。

4. `task model`
   在单次 run 上增加 excited-state 与 property task section，先提供真实最小实现和清晰 placeholder 边界。

## Architecture Choice

采用“run-centered aggregation”方案，而不是新起一套并行平台：

- 单点计算仍由 `workflow/runner.py` 驱动
- study/benchmark/scan 都围绕多个 `RunResult` 聚合
- excited-state/property 作为 `RunResult` 的正式 task section，而不是独立散落脚本
- capability/policy 在 config load 阶段和 result write 阶段双向固化

这种方案的优点是：

- 可以复用现有 artifact 与 benchmark 可信链路
- 不会把 schema 分裂成 run/study/scan 三套互不兼容结构
- exploratory 与 validated 的边界可以从单 run 一直传递到 aggregate report

## Validation Boundary

- validated：
  - exact/reference small-molecule baselines
  - existing statevector/shot benchmarked runs
  - dipole/property exact-expectation path on small exact tasks
- exploratory：
  - VQD-style excited-state interface skeleton
  - transition dipole / oscillator strength placeholders
  - runtime-ready and mitigation-ready integration points without remote runtime execution

## Deliverables

- Benchmark Suite v1
- Study / Registry / Campaign schema and artifact flow
- 1D scan workflow
- backend capability snapshot and policy layer
- excited-state + property task sections in run artifacts
- new aggregate reports and CLI entrypoints
