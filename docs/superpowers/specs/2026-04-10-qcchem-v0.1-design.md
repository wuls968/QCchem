# QCchem v0.1 Design

## 背景

用户要求把 QCchem 做成一个真正可扩展的量子化学软件雏形，而不是 notebook 拼接 demo。首版目标是建立一个最小但完整的 ground-state workflow，并明确 future-proof 的模块边界。

## 目标边界

本轮实现聚焦：

- 独立 Python package 与 CLI
- 自有统一数据模型
- YAML 驱动的分子输入与运行规格
- 电子结构问题构建
- Jordan-Wigner / Bravyi-Kitaev 映射
- 最小 ground-state 求解流程
- JSON/Markdown 工件、日志与 provenance
- H2、LiH、H2O(active-space) 示例与基础测试

不在本轮硬做的内容：

- 真机执行
- 完整 excited-state / property / geometry optimization 实现
- GUI

## 方案选择

### 方案 A：直接把 Qiskit Nature 与算法对象暴露到全项目

优点：实现快。

缺点：上层流程、CLI 与报告会被第三方对象形态绑死，后续替换 solver/backend 很痛。

### 方案 B：QCchem 自己维护统一 spec/result 模型，在边界层适配 Qiskit Nature 与 primitives

优点：架构更稳，结果工件统一，后续扩展容易。

缺点：首轮要多做一层抽象。

### 方案 C：先做脚本，后面再重构

优点：短期快。

缺点：直接偏离项目目标，可复现性与可维护性最差。

推荐采用方案 B。

## 关键设计

- `RunSpec` 作为工作流输入根对象，把 molecule/problem/mapping/backend/solver/run 聚合到一起
- `ElectronicStructureContext` 承载 problem 构建结果与化学摘要
- `MappingResult` 只负责从 fermion 到 qubit 的转换与统计
- `BaseSolver` 由 QCchem 定义，具体求解器只依赖 backend 抽象和 qubit operator
- `RunResult` 作为唯一报告数据源，JSON 与 Markdown 都从它派生

## 数据流

`config.yaml -> RunSpec -> ElectronicStructureContext -> MappingResult -> SolverResult -> RunResult -> artifacts`

## 测试策略

- 单元测试覆盖配置解析、结果序列化、映射统计
- 集成测试覆盖 H2 最小闭环
- 真实运行生成 artifact 作为 smoke proof

## 约束与假设

- 使用已有 conda 环境 `qiskit`
- 如果该环境缺少 `qiskit-nature` / `pyscf`，允许补装到同一环境
- 默认使用 statevector primitives V2 后端
- 活性空间能力先做规格与基本衔接，不追求复杂自动化
